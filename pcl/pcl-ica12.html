Three concepts the tutorial quietly hinges on

Tibble columns can be lists, and that’s where the power lives. A regular column is a vector — every cell is a single value. A list-column is a list — every cell can be a whole object: a sub-tibble, a fitted model, a ggplot, a list of polygon coordinates. Once you have a list-column, you can use mutate() with map() to apply any function to each cell, as if it were a row-level operation on objects more complex than scalars. This is the move that makes “fit a separate model for each group” into a single pipeline rather than a for loop with manual list management. The tutorial spends a lot of time on this because it is the purrr leverage point: everything else is supporting infrastructure.

map_* choice is about output type, not capability. map(xs, mean) returns a list of length-1 numbers — you’d need to unlist() it to plot or summarize. map_dbl(xs, mean) returns a numeric vector directly. The tutorial keeps using map_df to build summary tables row-by-row, with .id to preserve the iteration name. The base R analogue is the difference between lapply (always list) and vapply (typed and length-checked) — same design intent, friendlier syntax. Picking the right variant saves a step downstream every time.

map2 is paired iteration, not Cartesian product. This catches almost everyone once. If continents = c("Asia", "Americas") and years = c(1952, 2007), map2(continents, years, plot_fn) produces two plots: Asia 1952 and Americas 2007. It does not produce four plots covering all combinations. To get the Cartesian product you’d build the combinations first (expand_grid() or crossing()) and then pmap over the resulting tibble. The pattern data |> distinct(group_a, group_b) followed by pmap() is how you safely iterate over real combinations that exist in your data.

Two purrr patterns worth memorizing

The “model per group” pipeline. Probably the highest-leverage pattern in the entire tutorial:

gapminder |>
  group_by(continent) |>
  nest() |>
  mutate(
    model    = map(data, \(d) lm(lifeExp ~ pop + gdpPercap + year, data = d)),
    tidy_out = map(model, broom::tidy)
  ) |>
  ungroup() |>
  transmute(continent, tidy_out) |>
  unnest(tidy_out)

This produces one tidy tibble with one row per continent × term, with estimates, standard errors, p-values — ready for ggplot or gt. The shape generalizes: anywhere you’d otherwise loop over groups, fit something, extract something, and bind back, the nest → mutate(map) → unnest pipeline does it in one breath.

reduce() for repeated binary operations. When you need to apply the same binary operation across a list — joining many tables on a common key, row-binding many tibbles, intersecting many sets — reduce() is the right tool:

list_of_tables |> reduce(left_join, by = "id")     # join all on "id"
list_of_tibbles |> reduce(bind_rows)               # row-bind all
list_of_logical_vectors |> reduce(`&`)             # element-wise AND

The pattern is “fold the list into a single result by repeatedly applying a binary operation.” accumulate() is the same thing but keeps the intermediate results, useful for cumulative diagnostics.

Why this is the synthesis week

ICA 11 introduced purrr through the three R4DS use cases. ICA 12 generalizes: purrr is not a set of recipes, it’s a small algebra. map_* chooses your output type. map2 / pmap extend to multiple inputs. keep / discard / every / some answer logical questions of a list. reduce collapses a list to a single object. List-columns turn dplyr into a tool that operates on arbitrary objects, not just scalars. Once these compose, you stop thinking in loops and start thinking in transformations — what shape do I have, what shape do I want, what’s the function that gets me there?

A note on what this enables in practice

This is the layer that made several pieces of my STAT 212 work tractable. HW07 P1’s 196k JSON files: map(paths, jsonlite::read_json) |> list_rbind(). HW07 P2’s per-year ACS pulls: map(years, ~ get_acs(year = .x, ...)) |> list_rbind(names_to = "year"). HW08 P1’s pagination through cheese.com: map(1:21, scrape_page) |> list_rbind(). The TidyTuesday CNPJ analysis nests by legal_nature and computes Lorenz curves per nested sub-tibble — exact instance of the model-per-group pattern, but with a custom function instead of lm. None of these were “I wrote a loop and waited” — they were map, with the right output type, and a binary reduce or list_rbind at the end. The tutorial gives you the vocabulary; the work is recognizing the pattern.

Closing thought

The tutorial’s last section title is Now go forth and purrr. Worth taking literally: every time you find yourself reaching for a for loop in R, ask whether the operation has the shape “do the same thing to every element of a collection” — and if so, what the output type is. If the answer is “list,” use map; if “numeric vector,” use map_dbl; if “data frame stacked from each step,” use map_df or map(...) |> list_rbind(). Functional programming in R becomes muscle memory after maybe 20 deliberate uses; the loop instinct fades.
