Three concepts these readings quietly hinge on

A function’s last expression is its return value, automatically. R doesn’t require return(x) at the end of a function — whatever expression is last in the body is what gets returned. return() is reserved for early exit, when you want to bail out of the function before reaching the end. Idiomatic R style uses implicit return for the normal path and explicit return() only for guard clauses (if (is.na(x)) return(NA) at the top). Newcomers from Python or JavaScript often write return(result) on the last line out of habit; it works, but it’s noise.

if takes a single TRUE/FALSE; if_else() (or case_when()) takes a vector. This is the most common control-flow confusion in R. Inside a function with scalar logic — “if the input is missing, do X” — use if (is.na(x)) .... Inside a mutate() where you’re branching on a column — “for each row, if the value is positive, do Y” — use if_else(col > 0, ...). Mixing them up gives one of two errors: if complains about a vector condition, or if_else() silently treats your scalar as a length-1 vector and surprises you with the result. Peng 13.1 covers if; the dplyr equivalents are the vector versions.

Variables inside a function don’t leak out, but variables outside do leak in. This is lexical scoping. If a function references a name it didn’t define, R looks for that name in the enclosing scope (typically your global workspace). This is convenient — you can use packages without passing them in — but it’s also dangerous: a function that accidentally relies on a global variable will work in your console and silently break for someone else. Defensive practice: a function should take everything it depends on as an argument (data, columns, parameters), not reach into the global environment for them.

A small decision tree for function design

Question Default move
Does this code repeat with small variations? Refactor into a function
What goes in front of the parens? A verb in snake_case, distinct from base R names
First argument? The data — so the function pipes well
What kind of function is this? Vector (returns vector), data frame (returns tibble), plot (returns ggplot)
Does it take column names from a tibble? Use { col } for tidy-eval embracing
Should it have side effects? Avoid if possible — pure functions compose better
What edge cases to test? Empty input, single element, all NA, wrong type
Branching on scalar input? if/else if/else
Branching on a vector inside mutate()? if_else() or case_when()

Why this comes after wrangling

ICA 09 sits at the inflection point of STAT 212. Up to ICA 08, everything has been use of existing tools — call mutate(), call geom_point(), call left_join(). From ICA 09 onward, you’re making tools. The skills compose: a wrangling pipeline you’d otherwise repeat across 5 datasets becomes a single function called 5 times. A plot specification you’d otherwise duplicate across 10 facets becomes a function returning a ggplot. This is the move from “user of dplyr” to “extender of dplyr” — and it’s also the prerequisite for ICAs 11-12 (loops and iteration), where you’ll apply functions across collections.

Why if-else gets its own reading

Peng 13.1 is short but central. Almost every nontrivial function has at least one if — to handle a missing argument, to branch on data type, to short-circuit on edge cases. R4DS Ch25 assumes you know if/else and focuses on function design; Peng fills in the syntax and scoping rules. Worth flagging: R’s if is an expression, not just a statement — meaning y <- if (cond) 10 else 0 is valid and idiomatic, where languages like Python require a separate ternary syntax (y = 10 if cond else 0). Treating if as an expression makes function bodies cleaner: assignments at the top, conditional logic in the middle, the result naturally falling out at the bottom.
