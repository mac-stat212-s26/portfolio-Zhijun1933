Three concepts the chapter quietly hinges on

[ returns the same type; [[ drills down a level. The single most-confusing thing about R lists. l[1] returns a list of length 1 — the same type, smaller. l[[1]] returns whatever was at position 1 — a level deeper. The pepper-shaker analogy R4DS uses is exactly right: pepper[1] is a one-packet shaker, pepper[[1]] is the packet’s contents. The reason this matters: most operations you actually want to do (call a function on the contents, print the value, do arithmetic) need [[, not [. [ is for slicing collections; [[ is for unpacking a single item. New users who reach for [ everywhere end up with mysteriously nested results.

Tibbles are “lazy and surly”; data.frames are silently helpful. R4DS’s joke captures a real safety property. data.frame does partial matching on $: df$x works even when the actual column is x_value, returning the matching column without warning. tibble refuses partial matches and warns when you ask for an unknown column. This means switching from data.frame to tibble will surface bugs that were silently working — typos in column names that happened to match a prefix of a real column. This is a feature, not a regression.

Iteratively growing a vector is what makes “for loops slow” — not the loop itself. The myth that R’s for is slow is mostly wrong. What’s slow is the anti-pattern out <- c(out, new_value) inside a loop, because each iteration allocates a new vector and copies the old one. The fix: pre-allocate the output once (vector("list", n) or numeric(n)) then assign by index (out[[i]] <- ...). With pre-allocation, for is comparable in speed to lapply() and purrr::map(). The base-R apply family and tidyverse map family aren’t faster because of some clever trick — they’re faster because they pre-allocate for you.

Tidyverse ↔︎ base R cheatsheet

A small lookup for reading other people’s code:

What you want Tidyverse Base R
Filter rows by condition df \|> filter(x > 1) df[!is.na(df$x) & df$x > 1, ]
Sort rows df \|> arrange(x, y) df[order(df$x, df$y), ]
Select columns df \|> select(x, z) df[, c("x", "z")]
Filter and select together df \|> filter(...) \|> select(...) subset(df, ..., select = ...)
Pull a single column as vector df \|> pull(x) df$x or df[["x"]]
Add a new column df \|> mutate(z = x + y) df$z <- df$x + df$y
Map a function over a list purrr::map(xs, f) lapply(xs, f)
Map and simplify purrr::map_dbl(xs, f) vapply(xs, f, numeric(1))
Grouped summary df \|> group_by(g) \|> summarize(m = mean(x)) tapply(df$x, df$g, mean)
Side-effect over a vector purrr::walk(xs, f) for (x in xs) f(x)
Quick scatter for exploration qplot(x, y) or ggplot(...) + geom_point() plot(df$x, df$y)
Quick histogram ggplot(...) + geom_histogram() hist(df$x)

Why this comes after functions and before iteration

ICA 10 sits between Functions (ICA 09) and Loops/Iteration (ICA 11-12). The placement is deliberate: the apply family and for loops are base R’s iteration primitives, and ICA 11-12 will introduce purrr’s map_* family as the tidyverse alternative. By covering base subsetting and apply first, ICA 10 ensures you can read the iterating-with-base-R style you’ll inevitably encounter — and crucially, you’ll understand that purrr’s map functions are not magic: under the hood they’re doing what lapply() does, just with consistent return types and progress bars.

A note on “use what works”

The chapter’s most important meta-message is the line “It’s 100% okay to write code that uses a mix of approaches, and don’t let anyone tell you otherwise!” The tidyverse-vs-base-R debate is mostly a community culture artifact, not a technical one. Production R code regularly mixes both — using map() for the iteration shape and [[ for fast extraction, using filter() for the readable chain and df$col for a quick interactive check. Fluency means knowing both vocabularies and choosing the clearer one for each piece of code. ICA 10 is what makes the second vocabulary legible.

## Tidyverse ↔ base R cheatsheet

A small lookup for reading other people's code:

| What you want | Tidyverse | Base R |
|---------------|-----------|--------|
| Filter rows by condition | `df \|> filter(x > 1)` | `df[!is.na(df$x) & df$x > 1, ]` |
| Sort rows | `df \|> arrange(x, y)` | `df[order(df$x, df$y), ]` |
| Select columns | `df \|> select(x, z)` | `df[, c("x", "z")]` |
| Filter and select together | `df \|> filter(...) \|> select(...)` | `subset(df, ..., select = ...)` |
| Pull a single column as vector | `df \|> pull(x)` | `df$x` or `df[["x"]]` |
| Add a new column | `df \|> mutate(z = x + y)` | `df$z <- df$x + df$y` |
| Map a function over a list | `purrr::map(xs, f)` | `lapply(xs, f)` |
| Map and simplify | `purrr::map_dbl(xs, f)` | `vapply(xs, f, numeric(1))` |
| Grouped summary | `df \|> group_by(g) \|> summarize(m = mean(x))` | `tapply(df$x, df$g, mean)` |
| Side-effect over a vector | `purrr::walk(xs, f)` | `for (x in xs) f(x)` |
| Quick scatter for exploration | `qplot(x, y)` or `ggplot(...) + geom_point()` | `plot(df$x, df$y)` |
| Quick histogram | `ggplot(...) + geom_histogram()` | `hist(df$x)` |
