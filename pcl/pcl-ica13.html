Three concepts these readings quietly hinge on

JSON is just nested name-value pairs and arrays — but the nesting matters. A flat JSON response (single name-value pairs at top level) parses straight into a one-row tibble. A list of objects (an array where each element is itself a name-value object — like the results array in the Scott Lowe example) parses into a tibble with one row per element, after unnest_wider. A nested object inside another object requires another unnest_wider. Most real-world API responses combine both shapes, and reading JSON well is mostly about recognizing which shape you have at each level. The Mia Museum data from HW07 is the canonical case: each artwork is an object, and inside it creator is itself an array of nested objects (one per artist).

A URL is a structured address, not a string — and treating it like a string is the source of API bugs. When you concatenate "https://api.example.com?year=" + year, you’re treating the URL as a string. This works until the year is "2020 (revised)" or the API needs additional parameters that may or may not be present. The structured view is: a URL has six fields (scheme, domain, port, path, query, fragment), and any one of them can change independently. urltools::url_parse() materializes that structure as a data frame; param_set() and param_remove() modify the query without rebuilding the whole string. Once you think of URLs as objects rather than strings, building requests dynamically becomes safe.

Encoding is what keeps your URL parsable when values contain reserved characters. A query parameter value containing &, ?, /, =, or a space will break the URL if not encoded. The reserved-character problem is silent: the request often appears to succeed but returns wrong or empty data because the API parsed the URL differently than you intended. The fix: percent-encode the value with url_encode() (or let httr2 handle it automatically — it does). The principle: encode values, never the separators. Base R’s URLencode(reserved = TRUE) over-encodes by default and is the wrong tool here. Worth memorizing: space is %20, slash is %2F, ampersand is %26.

A small reference for URL anatomy

The six pieces, with examples:

Component Example Notes
scheme https What protocol — http, https, ftp, file
host api.census.gov Domain name or IP
port :8080 (omitted means default — 80 for http, 443 for https) Rarely needed for public APIs
path /data/2020/acs/acs1 Identifies the resource on the host
query ?get=NAME,B01001_001E&for=state:* Key-value parameters after ?, separated by &
fragment #section-3 Browser-side anchor; rarely matters for APIs

Two patterns that compose JSON + URL + urltools

Building a parameterized URL programmatically. Rather than concatenating strings:

base <- "https://api.census.gov/data/2020/acs/acs1"
url <- base |>
  urltools::param_set("get", "NAME,B01001_001E") |>
  urltools::param_set("for", "state:*") |>
  urltools::param_set("key", api_key)

Each param_set() call is independent and order-insensitive. Compared to paste0(base, "?get=NAME&for=state:*&key=", api_key), this is robust against missing values, accidental double &, and special characters that need encoding.

Parsing a JSON response into a tidy tibble. A typical response shape — array of objects with some nested fields:

resp <- httr2::request(url) |> httr2::req_perform()
data <- httr2::resp_body_json(resp)

# data is a list of lists. Convert to tibble:
tibble(record = data) |>
  unnest_wider(record) |>
  unnest_longer(authors)   # if 'authors' is an array

unnest_wider flattens a column of lists-of-named-things into multiple columns (one per name). unnest_longer expands an array column into multiple rows. Together they tame almost any JSON shape into a tabular one.

Why this comes before the actual API call

ICA 14 will introduce httr2 and the request/response cycle. ICA 13 builds the foundation: by then you should already know what shape the data will arrive in (JSON), what form the request takes (URL with query parameters), and how to manipulate either programmatically. Without these, you’d be debugging string concatenation and re-learning JSON anatomy at the same time as learning HTTP verbs and authentication. Splitting the prep this way keeps each ICA’s cognitive load reasonable.

A note on what this enables

The three building blocks here are exactly what HW07 P2 needs. The Census ACS endpoint is a URL with query parameters — param_set() is how you swap years across calls. The response is JSON arrays — fromJSON() plus unnest_wider() flattens them. And the API key is just one more parameter to set. The pattern, once seen, is portable: every JSON-over-HTTP API you’ll meet (OpenWeather, Spotify, GitHub, Twitter, NYT) follows this shape. ICA 13 gives you the vocabulary; ICA 14 gives you the verb.
