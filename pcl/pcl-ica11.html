Pre-class learning map

R4DS Chapter 26 is where ICA 09 (writing functions) and ICA 10 (base R primitives) fuse into a workflow. Once you can write a function, the next question is: how do I apply it to a hundred things? The chapter answers this with three concrete tasks rather than abstract theory: modifying multiple columns of one data frame (across), reading multiple files into one data frame (map + list_rbind), and writing one R object out to multiple files (walk / walk2). Each task is a slightly different shape of the same idea — functional programming means passing functions around so a higher-order function can do the looping for you — and each unlocks a class of analysis that copy-paste cannot reach. Three concepts the chapter quietly hinges on

Pass the function, don’t call it. The single most common error inside across() and map(): writing across(a:d, median()) instead of across(a:d, median). The first calls median() immediately with no input, which errors. The second passes the function median itself to across, which then calls it on each column. The mental shift: in functional programming, function names are values you can hand around. The parentheses are how you invoke a function; without them, you’re referring to the function as a thing. This is the same logic as the difference between a recipe and cooking — you give across the recipe, it does the cooking.

map() returns a list; the typed variants return vectors. map(xs, f) always returns a list, even if every result is a single number. To get a numeric vector, use map_dbl(xs, f). To get a character vector, use map_chr. To get logical, map_lgl. The choice matters because lists are slow to work with downstream — purrr::map_dbl(xs, mean) gives you a vector you can plot and summarize directly, while map(xs, mean) gives you a list you’ll have to unlist() first. The base R analogue is lapply() (list) vs vapply() (typed); the design intent is the same.

walk() is map() with the result thrown away — and that’s a feature. When iteration is for side effects (writing files, appending to a database, printing progress), the return value is uninteresting noise. walk(paths, append_file) does the same thing as map(paths, append_file) but returns the input invisibly so you can keep piping. Using map for side effects works but produces a list of NULLs or whatever the side-effecting function returned, which clutters your workspace and obscures intent. The walk family signals: “I’m doing this for the effect, not the value.”

Two iteration design choices worth flagging

One complex step vs many simple steps. When you need to read 12 files and clean each one, R4DS recommends the many simple steps approach: map(read_csv) |> map(filter) |> map(mutate) |> list_rbind() rather than map(process_file) where process_file does everything. The reasoning: writing one big function tempts you to fixate on getting file 1 right before looking at the others, when in reality you should be looking at all of them holistically. Many simple steps lets you see the full collection at every stage. Even better when possible: bind the data frames before cleaning and let regular dplyr handle the rest.

The set_names() move for path data. When the filename itself contains data (year, region, version, …), set_names(basename) carries that filename forward as list-element names, then list_rbind(names_to = "year") materializes it into a column. This is the cleanest way to recover information from the directory structure without writing a custom parser. For more complex paths (multiple data fields embedded), set_names() with no argument keeps the full path, then separate_wider_delim() parses it.

Why this comes after functions and base R

ICA 11 is the synthesis week. ICA 09 taught how to write a function; ICA 10 taught how lapply and for iterate in base R; ICA 11 introduces purrr’s map_* family as the more consistent, more pipe-friendly version of the same idea. The progression is intentional: by the time you meet map(), you know that under the hood it’s doing what lapply() does, that the function you pass is one you might have just written yourself, and that the value of doing this — over a for loop — is composability with the rest of your tidyverse pipeline. Functional programming becomes legible because all three pieces (functions, base iteration, tidyverse iteration) sit in one connected mental model.

A note on what this enables

The skills in this chapter are what make HW07 and HW08 possible. HW07 P1 reads ~196k JSON files and binds them — that’s list.files() |> map(read_json) |> list_rbind() at scale. HW07 P2 hits the Census API repeatedly across years — that’s map(years, fetch_year) |> list_rbind(names_to = "year"). HW08 P1 scrapes 21 alphabetical pages from cheese.com — that’s map(1:21, scrape_page) |> list_rbind(). None of these are realistic without iteration. The pattern, once internalized, is the same shape every time: list the inputs, map the work function over them, bind the results.
