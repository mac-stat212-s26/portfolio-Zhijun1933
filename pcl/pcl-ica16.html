Pre-class learning map

R4DS Chapter 21 takes an unusual pedagogical route: it teaches SQL backwards through dbplyr. Rather than introduce SQL from scratch (SELECT col FROM table WHERE ...), it shows that the dplyr verbs you already know (select, filter, arrange, group_by, summarize, *_join) translate one-to-one into SQL clauses, and that show_query() will print the SQL dbplyr generates. The benefit is that SQL stops feeling like a foreign language: it’s the output of a dplyr pipeline, just written in a different syntax. Three concepts the chapter quietly hinges on

SQL writes in one order, evaluates in another. The clauses are written SELECT ... FROM ... WHERE ... GROUP BY ... ORDER BY, but they’re evaluated in order FROM → WHERE → GROUP BY → SELECT → ORDER BY. This mismatch has a real consequence: you can’t reference a column you create in SELECT from inside WHERE, because WHERE runs first. dplyr hides this — mutate(x2 = x + 1) |> filter(x2 > 5) looks straightforward — but dbplyr handles it by generating a subquery: an inner SELECT that creates the new column, wrapped by an outer SELECT that filters on it. Reading dbplyr’s show_query() output for such cases is the fastest way to internalize why subqueries exist.

Lazy evaluation is what makes dbplyr useful at scale. When you call tbl(con, "flights") |> filter(...) |> select(...), nothing actually runs. dbplyr just records the operations as a query plan. The plan executes only when you print it (with a LIMIT for the preview), call collect() (pull all rows into R), or call compute() (materialize as a temporary table on the server). The win: you can compose a 12-step pipeline against a 100-million-row table, and only when you collect() does the database actually run anything — and even then it’s the result that comes back, not the underlying table. Without laziness, every intermediate step would round-trip through R, defeating the whole point of using a database.

NULL is not NA — and it’s silently dropped in aggregations. R’s NA is “infectious”: NA + 1 is NA, mean(c(1, 2, NA)) is NA. SQL’s NULL follows three-valued logic in expressions (also infectious there) but is silently removed from aggregations: AVG(), COUNT(), SUM() all skip NULLs by default. dbplyr issues a one-time warning when you first hit this — “Missing values are always removed in SQL aggregation functions” — but it’s worth knowing in advance. Practical consequence: a mean(arr_delay) in dplyr-on-database returns the average of non-missing values automatically, whereas the same in dplyr-on-tibble returns NA unless you write na.rm = TRUE. If you need to know how many were missing, count them separately with sum(is.na(x)).

A small reference for dplyr ↔︎ SQL translation

The core mapping you’ll re-derive every time you read someone else’s code:

dplyr SQL clause Notes
select(col1, col2) SELECT col1, col2 Columns chosen
rename(new = old) SELECT old AS new Aliasing — old on left in dplyr, in SQL
mutate(z = x + y) SELECT *, x + y AS z New column expression
filter(x > 0) WHERE x > 0 Pre-aggregation filter
filter(grp_n > 5) after summarize HAVING ... > 5 Post-aggregation filter
arrange(x) / arrange(desc(x)) ORDER BY x / ORDER BY x DESC Sort
group_by(g) %>% summarize(...) GROUP BY g + aggregate in SELECT Aggregation
left_join(b, by = "k") LEFT JOIN b ON a.k = b.k Join
mean(x, na.rm = TRUE) AVG(x) NULL auto-skipped
n() COUNT(*) Row count
if_else(cond, a, b) CASE WHEN cond THEN a ELSE b END Branch
lead(x) / lag(x) LEAD(x) OVER (...) / LAG(x) OVER (...) Window function
== = SQL has only one
& | ! AND OR NOT Spelled out
%in% c('a','b') IN ('a', 'b') Set membership
is.na(x) x IS NULL Null check

Two patterns worth memorizing

The dbplyr workflow shape. This is the canonical pipeline for using a database from R:

con <- DBI::dbConnect(duckdb::duckdb(), dbdir = "data/my.duckdb")

# Reference, doesn't load anything
flights_db <- tbl(con, "flights")

# Compose a query lazily
delayed_flights <- flights_db |>
  filter(arr_delay > 60) |>
  group_by(dest) |>
  summarize(n = n(), avg_delay = mean(arr_delay, na.rm = TRUE)) |>
  arrange(desc(n))

# Inspect what would run
delayed_flights |> show_query()

# Pull results into R when ready
result <- delayed_flights |> collect()

DBI::dbDisconnect(con)

Connect → reference → compose → inspect → collect. Doing it in this order means you only pull computed results, never the underlying table.

The “see what SQL it generates” trick for learning. The fastest way to learn SQL given dplyr fluency:

flights_db |>
  filter(dest == "IAH") |>
  group_by(carrier) |>
  summarize(avg_delay = mean(arr_delay, na.rm = TRUE)) |>
  arrange(desc(avg_delay)) |>
  show_query()

Read the output. Match each line to a dplyr verb. After 10 such pipelines, the SQL clause vocabulary becomes familiar. This is how the chapter teaches; it’s also how to keep learning whenever you hit a new pattern.

Why this comes after iteration and APIs

ICA 16 sits at the end of the data-acquisition arc. ICAs 13-14 taught APIs (request data over HTTP), ICA 15 taught scraping (extract data from HTML), ICA 16 teaches databases (query data from a structured store). The progression is from external + ad-hoc to external + structured to internal + structured. By this point, you have three orthogonal ways to acquire data, each with a different cost-benefit profile: APIs are clean but rate-limited and may not exist; scraping is universal but fragile; databases are powerful but require infrastructure. Knowing all three means you can usually find a path to whatever data you need.

A note on what this enables

HW08 P2 is exactly this pattern, except the database is run by someone else (Stack Exchange Data Explorer). Each of the four queries — users, badges, posts, comments — is a SELECT ... FROM ... WHERE ... [GROUP BY ...] written by hand in SEDE’s web interface, downloaded as CSV, then loaded into R. The structure is identical to dbplyr’s output: pick columns, filter rows, group and aggregate where needed, and SEDE returns the result. The reason that pattern works at billion-row scale (the Stack Overflow dump is genuinely massive) is exactly what this chapter is about: the database engine does the heavy lifting before any data crosses the wire.

A small philosophical note

The chapter ends with: “it’s important to learn some SQL because it’s the most commonly used language for working with data and knowing some will make it easier for you to communicate with other data folks who don’t use R.” This is true and worth taking seriously. SQL was designed in the 1970s and has outlasted nearly every other data-tool fashion. The dialect drift between Postgres, MySQL, BigQuery, and Snowflake is real but the core 80% — SELECT / FROM / WHERE / GROUP BY / JOIN / window functions — is portable, durable, and present in every data infrastructure stack you’ll meet. dbplyr is a wonderful learning tool and a real production tool, but the underlying skill it teaches — thinking in queries — is what carries forward.

## A small reference for dplyr ↔ SQL translation

The core mapping you'll re-derive every time you read someone else's code:

| dplyr | SQL clause | Notes |
|-------|-----------|-------|
| `select(col1, col2)` | `SELECT col1, col2` | Columns chosen |
| `rename(new = old)` | `SELECT old AS new` | Aliasing — old on left in dplyr, in SQL |
| `mutate(z = x + y)` | `SELECT *, x + y AS z` | New column expression |
| `filter(x > 0)` | `WHERE x > 0` | Pre-aggregation filter |
| `filter(grp_n > 5)` after summarize | `HAVING ... > 5` | Post-aggregation filter |
| `arrange(x)` / `arrange(desc(x))` | `ORDER BY x` / `ORDER BY x DESC` | Sort |
| `group_by(g) %>% summarize(...)` | `GROUP BY g` + aggregate in `SELECT` | Aggregation |
| `left_join(b, by = "k")` | `LEFT JOIN b ON a.k = b.k` | Join |
| `mean(x, na.rm = TRUE)` | `AVG(x)` | NULL auto-skipped |
| `n()` | `COUNT(*)` | Row count |
| `if_else(cond, a, b)` | `CASE WHEN cond THEN a ELSE b END` | Branch |
| `lead(x)` / `lag(x)` | `LEAD(x) OVER (...)` / `LAG(x) OVER (...)` | Window function |
| `==` | `=` | SQL has only one |
| `&` `|` `!` | `AND` `OR` `NOT` | Spelled out |
| `%in% c('a','b')` | `IN ('a', 'b')` | Set membership |
| `is.na(x)` | `x IS NULL` | Null check |
