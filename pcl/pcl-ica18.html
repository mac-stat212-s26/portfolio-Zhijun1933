Pre-class learning map

ICA 18’s two readings are deliberately in tension with each other. Cole Nussbaumer Knaflic’s Storytelling with Data podcast argues for the storytelling tradition: a curated narrative, a focused chart, annotations that point the reader to the takeaway. Dominikus Baur’s Medium post The death of interactive infographics? presents a sobering empirical fact from the New York Times: only 10–15% of readers click on essential buttons, and 85% never see anything hidden behind interaction. The pair forces the question that ICA 18 is really about: when does interactivity earn its complexity, and when is a static, well-annotated chart actually better? The answer hinges on three audience variables — time, goals, and care — that determine whether interaction is a feature or a barrier. The mindmap organizes the prep around the storytelling argument, the empirical critique, and the three-axis decision framework that reconciles them.

Three concepts these readings quietly hinge on

The 85% statistic should change how you design. Aisch’s number — that only 10–15% of NYT readers click on interactive elements — is not just sobering; it should reshape decision priors. A reasonable mental model: whatever you put behind a click, 85% of your audience will never see. This means the default state of an interactive piece must already convey the core message; interaction is only ever for more, not for the point. Baur’s quote of Tse — “If you make a tooltip or rollover, assume no one will ever see it” — is the operationalization. Tooltips can hold supplementary detail (data values, sources, footnotes), but never the headline. The headline goes in the title, the annotation, the static layer.

Time + Goals + Care is the three-axis filter for interactivity. Baur’s framework names the variables that actually determine whether interaction works. Time: a commute reader has 30 seconds and will not click; a museum visitor at an installation has 30 minutes and may. Goals: a reader learning about a topic wants a guided narrative; a reader looking for a specific answer (where on my line can I get cell signal?) needs a tool. Care: a reader’s interest must be activated before they engage — through personalization (defaulting to their location), through narrative hook (why this matters to you), or through onboarding (showing them how to read it). When all three favor interaction, build it; when any one fails, default to static. This collapses years of data-vis design debate into a tractable triage.

Storytelling and exploration are different jobs, not opposing philosophies. Knaflic and Baur appear to disagree, but they’re optimizing for different contexts. Knaflic’s audience: a manager presenting a quarterly result to executives, a journalist with a single point to communicate, an analyst justifying a recommendation. The job is persuasion — one message, well-told. Baur’s audience: a researcher exploring an unfamiliar dataset, a museum-goer at an installation, a transit rider answering “where can I get signal.” The job is discovery — many possible messages, the reader picks. The bug is treating these as the same job. A persuasive chart with too many interactions dilutes its message; an exploratory tool with a forced narrative removes the agency that justified building it. Match the form to the job.

A small reference for the decision

Before adding any interactivity, three questions in order:

Question If “static” If “interactive”
How long will the reader spend? Brief / scrolling Sustained / focused
Is there one core takeaway? Yes — annotate it No — multiple meaningful views
Will the reader know what to do? Reading is the action Needs onboarding or sensible defaults
Is the goal persuasion? Yes — story arc Goal is exploration or tool-use
Will the chart appear in feed / mobile? Yes — assume scroll No — captive context

If two or more rows lean static, default to static + annotation. If two or more lean interactive, build interaction with onboarding.

Two patterns worth memorizing

The “default state must carry the message” pattern. Even when building an interactive, design as if interaction will not occur:

# Bad: the headline is hidden behind a tooltip
plot_ly(data, ...) |>
  add_markers() |>
  add_text(text = ")          # nothing visible by default

# Good: headline annotated on the static layer; interaction adds detail
ggplot(data, aes(x, y)) +
  geom_point(aes(color = important_flag)) +
  geom_text(data = highlights,
            aes(label = label),
            color = "firebrick") +                 # visible without click
  labs(
    title = "Headline conclusion in plain text",
    subtitle = "Source data and detail available on hover"
  ) |>
  plotly::ggplotly()                                # interactivity layered on

The static version of an interactive should still tell the story. Interaction adds depth, never the substance.

The personalize-first pattern. When interactivity is justified, hook the reader by starting from their data:

# OECD pattern: detect or ask, default to user's region
user_region <- detect_or_default_region()  # via geolocation or dropdown

oecd_data |>
  filter(region == user_region) |>
  plot_initial_view()                       # this is what they see first

# Then make exploration available
ui_for_changing_region(default = user_region)

The reader arrives in a chart that’s already about them; agency to change the view is offered, not required. This converts the “why should I care” question before the reader can ask it.

Why this comes after effective viz

ICA 17 taught the design discipline of static charts — data-to-ink, references, highlighting, ordering, narrative arc. ICA 18 takes those principles and asks: which of them change when you add interaction? Some carry over directly (high data density, meaningful order), some get inverted (tooltips can violate “highlight one thing” because the user picks), and some become harder (small multiples in interactive context need careful onboarding). The progression reflects how the tools are actually built: plotly::ggplotly() adds interaction on top of a well-designed ggplot, so the static design discipline is the foundation, not the alternative.

A note on what this enables

The decision framework here directly shapes how I’ll build the final TidyTuesday entries and any portfolio dashboards. For most TidyTuesday posts — single dataset, single weekly question, scrolling reader — static + annotation wins; the time investment in interactivity isn’t repaid by the audience. For an exploratory piece (the CNPJ Lorenz analysis is a candidate) the interactive version might be justified — but only if the static version still carries the headline finding. The quiet rule both readings converge on: interaction is for the second question, not the first. Make sure the first question has a static answer.

Closing thought

The Baur essay’s title — The death of interactive infographics? — is intentionally provocative. The honest answer is the one he gives: it depends. The maturation of the field means we’re past the period when interaction was a virtue in itself, and into a period when it’s a tool to be deployed only when its complexity is repaid by audience benefit. Knaflic’s storytelling tradition isn’t an alternative to interaction; it’s the discipline that lets interaction serve a purpose when you do choose to add it. ICA 18’s job is to internalize the question — what does interaction earn here? — and only build it when there’s an answer.
