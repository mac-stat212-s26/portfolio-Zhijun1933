Three concepts these chapters quietly hinge on

== is treacherous on floats; near() is what you actually want. The classic gotcha: sqrt(2)^2 == 2 returns FALSE in R because floating-point arithmetic introduces tiny rounding errors. The right pattern is near(sqrt(2)^2, 2) which checks equality within tolerance. R4DS Ch12 mentions this in passing but it’s the source of countless silent filter bugs — filter(df, value == 0.1) may drop rows where the value is 0.09999999999999998. The general rule: use == only on integers, characters, and factors; use near() on doubles.

Factors aren’t just labeled strings — they’re strings with a predetermined level set. This matters in two places: (1) ggplot uses factor levels as the x-axis order, so fct_reorder(species, body_mass) is how you sort a bar chart by something other than alphabetical; (2) case_when() with factor outputs needs all branches to return the same factor with the same levels, or you get NA for unmatched levels. The forcats package exists because manipulating levels (reorder, rename, collapse, lump) is much more common than people expect, and base R’s tools for it are awkward.

NA in logical operations follows three-valued logic, and the rules aren’t intuitive. NA & FALSE is FALSE (because regardless of what NA actually is, ANDing with FALSE gives FALSE). But NA & TRUE is NA (because if NA is FALSE, the answer is FALSE; if NA is TRUE, the answer is TRUE — we can’t decide). Same logic for OR: NA | TRUE is TRUE (always true), but NA | FALSE is NA. This is Kleene logic, not a bug. The takeaway: filtering with is.na() is explicit and safe; comparisons that produce NA get silently dropped by filter() because filter() treats NA as “not TRUE.”

The unifying move

All four chapters teach the same shape: a mutate() or filter() or summarize() call where the value of the verb’s argument is a type-specific function. The skeleton is always:

df |>
  mutate(new_col = TYPE_SPECIFIC_FN(old_col)) |>
  filter(LOGICAL_FN(some_col)) |>
  summarize(stat = NUMERIC_SUMMARY(some_col), .by = group_col)

What changes between chapters is which TYPE_SPECIFIC_FN you reach for:

Column type Common toolkit
Logical if_else, case_when, any, all
Numeric round, lag, cumsum, min_rank
Factor fct_reorder, fct_lump, fct_recode
Date/time year, wday, month, floor_date

The dplyr verbs are the spine; the type-specific functions are the muscle.

Why this comes before joins and pivots

ICA 06 is “Adv Wrangling I” — and by week 6 of STAT 212, the grammar from Ch1 + the basic wrangling verbs from Ch3-5 are already in muscle memory. What was missing was the toolkit for handling specific column types cleanly: how to bin a numeric variable, how to reorder a factor for plotting, how to extract day-of-week from a date column to detect weekly patterns. These four chapters fill in that gap, which makes the next steps (joins, pivots, missing data) workable on real-world data — where columns aren’t always nicely numeric and dates don’t parse themselves.
