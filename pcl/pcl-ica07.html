Three concepts these chapters quietly hinge on

Backslashes in regex are doubly escaped in R strings. To match a literal period in regex, the regex engine wants \. — but R strings interpret \. first and pass . to regex, which means “any character.” The right pattern in R is "\\.": R string parser reduces \\ to \, then regex engine sees \. and matches a literal period. This is the source of more wrangling bugs than any other regex issue. Use raw strings (r"(\.)", R 4.0+) when you have many backslashes — it suspends R-level escaping. Or test patterns in str_view() first, which prints what regex actually receives.

left_join() is almost always what you want; inner_join() is what you sometimes want. New users often default to inner_join() because it sounds tidier (“only matched rows”), but in practice it silently drops rows you wanted to keep. The diagnostic question is: which table is your “main” table? The main table goes on the left, you left_join() lookup tables onto it, and you check NA in the new columns to see what didn’t match. inner_join() is appropriate only when every row of both tables must have a match, which is rarer than it sounds.

Many-to-many joins multiply rows, and dplyr now warns you for a reason. If table A has 3 rows for id = 1 and table B has 4 rows for id = 1, joining them produces 12 rows for id = 1 — every combination. This is sometimes what you want (combining time-varying attributes), but most of the time it’s a sign that one of your “keys” isn’t actually unique. Check with count(df, key) |> filter(n > 1) before joining. Since dplyr 1.1.0 (2023), many-to-many joins emit a warning unless you pass relationship = "many-to-many" explicitly, which forces you to acknowledge the choice.

Strings + regex + joins, in one sentence

If the previous PCL was “what do you do with each column type,” this one is “what do you do when columns are messy strings and what do you do when the data you need is in another table.” Together with Ch12-13 (logical, numeric) and Ch16-17 (factor, date), the toolkit is now complete for almost any single-row-per-observation dataset that arrives in tabular form.

Practical heuristic for joins

A small decision tree that captures most wrangling-time reasoning:

Question Answer
Adding columns to your main table? mutating join (left_join, inner_join, …)
Filtering rows of your main table by another table’s contents? filtering join (semi_join, anti_join)
Want to keep rows that didn’t match? left_join (then NA will appear in new columns)
Want to drop rows that didn’t match? inner_join
Want to find rows that should match but don’t? anti_join
Both tables have a column with the same name as the key? by = "key"
Different names? by = c("a" = "b")
Worried about duplicate keys? count() first, then join with relationship = "..." set explicitly

Joins are the gateway from “single-table dplyr” to “real-world relational data.” The rest of STAT 212 — APIs returning nested JSON, web scraping that produces multiple coordinated tables, SQL — all assume you’re fluent in this paradigm.
