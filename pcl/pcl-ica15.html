Pre-class learning map

ICA 15 is the mirror image of ICAs 13–14. APIs work when the website’s owner has built one and made it accessible; web scraping works when they haven’t, or when the data is only available through the rendered page itself. The mechanics are different — instead of a clean JSON response keyed by parameter, you get a hierarchical HTML document and have to address into it using CSS selectors, the same mini-language CSS uses to apply styles. The rvest vignette teaches three things in sequence: (1) HTML structure as a tree of nested tags with attributes; (2) CSS selectors as the addressing language for picking nodes out of that tree; (3) rvest’s extraction functions for getting text, attributes, or tabular data out of selected nodes. The mindmap below organizes the prep around these three layers plus the practical observation pattern (html_elements() outer / html_element() inner) that keeps your variables aligned.

Three concepts the vignette quietly hinges on

The observation pattern is what keeps your data frame aligned. This is the single most important pattern in rvest, and the one new scrapers most commonly miss. If you call html_elements("b"), html_elements("i"), html_elements(".weight") directly on the page, you’ll get vectors of different lengths whenever any observation is missing one of those fields — and there’s no safe way to align them back. The fix: first select the observation rows with html_elements() (e.g., the <li> for each Star Wars character), then extract each variable from each row with html_element() (singular) — which returns NA when no match is found, preserving alignment. The whole rvest API is designed around this two-level pattern. Internalize it and most scraping bugs disappear.

html_text2() mimics what the browser shows; html_text() returns raw character soup. HTML ignores whitespace in source code and uses element structure to render text. html_text() returns the source with all its newlines, tabs, and indentation preserved — ugly and rarely what you want. html_text2() reconstructs what the rendered page would look like: paragraphs separated by blank lines, no leading whitespace, sentences joined sensibly. The default reach should always be html_text2(). The only time html_text() is correct is when you specifically want to preserve raw whitespace (which is almost never).

Attributes are how the data not visible on screen gets carried. When scraping links from a list, the visible text might be the article title — but the URL you actually want is in the href attribute of the <a> tag. html_attr(node, "href") is how you get it. Same with images (src), data attributes (data-id, data-price), and form inputs (value). Always check what’s in the attribute as well as what’s in the text — half of the useful data on a typical page is in attributes, not in visible content.

A small reference for CSS selectors

The selector forms ranked by frequency in real scraping work:

Selector Matches Example use
tag All elements with that tag p, a, td
.class Elements with that class .product-name, .price
#id The unique element with that id #main-content
tag.class Tag with that class a.external, div.card
parent child Descendant (any depth) ul li, .card a
parent > child Direct child only ul > li, nav > a
[attr] / [attr=value] Attribute presence / equality [href], [data-type=card]
[attr^=value] Attribute starts with [href^=https]
:first-child / :nth-child(n) Positional li:nth-child(odd)
selector1, selector2 Either of these (union) h2, h3

For most scraping tasks, the first four are sufficient. The rest are tools you reach for when the page’s structure is hostile.

Two patterns worth memorizing

The complete scrape-one-page function. Generalized from the rvest Star Wars example:

scrape_page <- function(url) {
  page <- read_html(url)
  rows <- page |> html_elements("li.character")  # observation selector

  tibble(
    name    = rows |> html_element("b")        |> html_text2(),
    species = rows |> html_element("i")        |> html_text2(),
    weight  = rows |> html_element(".weight")  |> html_text2() |> parse_number(),
    profile = rows |> html_element("a")        |> html_attr("href")
  )
}

The shape: read → select rows → tibble of per-variable extractions. Every variable is one column-line; every line is the same length because html_element (singular) returns NA on miss.

The paginated scrape. Same shape as the API pagination from ICA 14, but with read_html instead of fromJSON:

urls <- str_glue("https://example.com/items?page={1:21}")

all_items <- urls |>
  map(scrape_page) |>          # one tibble per page
  list_rbind()                 # bind into one

Add Sys.sleep(0.5) inside scrape_page if the site rate-limits or if politeness matters.

Why this comes after APIs

ICAs 13–14 taught the structured case: a documented endpoint, a JSON contract, parameters that the maintainer has agreed to support. ICA 15 teaches the unstructured case: no contract, you read the HTML the page rendered for human eyes and address into it via styling hooks the developers happened to use. The skill ordering matters because once you’ve internalized JSON paths and URL parameters, CSS selectors feel familiar — they’re just a different addressing language for a different data shape. The iteration patterns (map over pages, bind with list_rbind) are identical to the API case; only the per-page extraction logic changes.

A note on what this enables

HW08 P1 is exactly this pattern. cheese.com has 21 alphabetical pages, each listing 100 cheeses (the last has 46) — map(1:21, scrape_page) |> list_rbind() produced the 2046-row index. The detail-page version takes each cheese’s URL (extracted from <a href>), follows it, and pulls the milk / country / family / type / flavor fields by .summary-points class — same observation pattern, just one row per page instead of many. The same template will scrape essentially any catalog-style website: paginated index → detail pages → bind. The vocabulary rvest gives you is the toolkit; recognizing the catalog-detail pattern is what makes a scraping task tractable in 30 lines instead of 300.

A note on legality and politeness

Scraping is legal in the US for publicly accessible pages (Van Buren v. United States, 2021), but ethical scraping is its own discipline. Three rules of thumb: (1) check robots.txt and respect Disallow rules; (2) rate-limit yourself even when not asked — a Sys.sleep(0.5) between requests costs nothing and keeps you off the maintainer’s radar; (3) cache locally and never re-scrape what you already have. Most scraping problems with hosting providers come not from scraping itself but from scraping aggressively — hammering a small site at full speed will get your IP banned and may break their server. The rvest vignette doesn’t dwell on this, but it’s part of the practice.
