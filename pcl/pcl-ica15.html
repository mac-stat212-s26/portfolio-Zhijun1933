Three concepts the chapter quietly hinges on

Scraping is reverse-engineering, not parsing. When you call read_html("https://example.com"), you get back the HTML the server sent to your browser — which may differ from what the browser actually rendered if the page uses JavaScript to load data after the initial response. For static sites, rvest is straightforward; for JS-heavy sites, the data you want may not be in read_html()’s output at all, and you’d need a tool like RSelenium or chromote that runs a real browser. The clue: if html_elements() returns empty when you can clearly see the data in your browser, the page is JS-rendered.

CSS selectors are the contract between you and the page’s HTML structure. A selector like .post-title h2 tells rvest: “find every element with class post-title, then find every h2 inside those.” This works as long as the page’s HTML structure stays the same — but the moment the site redesigns, your selector breaks. This is the core fragility of scraping: there’s no contract, no API version, no deprecation warning. The page just changes and your code silently returns empty results or wrong data. Defensive scraping checks the result count: if html_elements(".post-title") returned 50 yesterday and 0 today, something changed.

Politeness is both ethical and practical. Rate-limiting your scraper (Sys.sleep(1) between requests) protects the site from your traffic and protects you from being IP-banned. Setting a User-Agent header that identifies you (e.g., "Zhijun He's research scraper, contact: zhe@macalester.edu") lets the site admin reach out instead of blocking. Reading robots.txt (e.g., https://example.com/robots.txt) tells you which paths the site explicitly disallows. Respecting these isn’t optional courtesy — it’s how the open web stays scrapeable for everyone.

A small reference for rvest verbs

Verb What it does
read_html(url) Download and parse the page
html_elements(css) Select all matching elements (vector)
html_element(css) Select first match (single element)
html_text2() Extract clean text, normalize whitespace
html_attr("href") Extract attribute value
html_table() Parse <table> directly to data frame

A code pattern for scraping multiple pages

library(rvest)
library(purrr)

scrape_one <- function(url) {
  Sys.sleep(1)  # politeness
  page <- read_html(url)
  tibble(
    title = page |> html_elements(".title") |> html_text2(),
    date  = page |> html_elements(".date")  |> html_text2()
  )
}

urls <- paste0("https://example.com/page/", 1:10)
results <- map_dfr(urls, scrape_one)

The shape: define a function that scrapes one page, sleep between calls, map over a list of URLs, bind results into a tidy frame. This pattern works for any paginated source.

Why this comes after APIs

ICAs 13-14 taught APIs — clean, structured, contracted ways to get data. ICA 15 teaches scraping — what you do when no API exists. The order matters: APIs first means scraping is positioned as the fallback, not the default. Always check if an API exists before scraping; the maintenance burden of scraping (selectors break, sites redesign, rate limits change) is high enough that API access is almost always worth using when available.

A note on what this enables

HW07’s Mia Museum data was a JSON dump (effectively API output). HW08’s cheese.com data was scraping — no API, just HTML pages. The contrast is the point: same goal (get structured data), different tools, dramatically different maintenance profiles. Scraping is powerful but fragile; treating it as the last resort, not the first instinct, is the rule that keeps your data pipeline maintainable.
