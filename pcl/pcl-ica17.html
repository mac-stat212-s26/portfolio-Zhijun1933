Pre-class learning map

ICA 17 marks the moment in STAT 212 when viz stops being about how to draw and starts being about how to communicate. ICA 03 introduced the grammar of ggplot2 — the syntax of layered plots. ICA 17 introduces the principles that decide whether the plot succeeds — whether someone reading it leaves with the insight you intended. Two readings split the cognitive load: Brent Dykes’s Data Storytelling makes the why argument — that data alone, however accurate, doesn’t change minds; only data + visuals + narrative does, because decisions are mediated by emotion as well as logic. Peng’s Customizing ggplot2 Plots gives the how — six concrete design guidelines (data density, labels, references, highlighting, small multiples, meaningful order) that translate Tufte/Few/Wainer’s writing into ggplot2 idioms you can apply directly. Three concepts these readings quietly hinge on

A statistic informs; a story persuades. Dykes’s most cited point is that brain scans of statistics versus stories show different patterns of activation: statistics light up Broca’s and Wernicke’s (language processing only), while stories light up additional motor, sensory, and emotional regions. The implication for data work: a finding that is statistically valid but narratively flat will be remembered, recalled, and acted on far less than one that is identical in numbers but shaped into “a small farmer in rural Mali…” rather than “12.3% of children…” This isn’t manipulation — it’s recognition that human cognition is dual-process, and a chart that engages only the analytical channel leaves the persuasive channel on the table. Every plot you make has a narrative dimension whether you intend one or not; the question is whether you’ve chosen what it says.

Data-to-ink ratio is the meta-rule that contains most others. Tufte’s principle — every drop of ink should be carrying information — is the geometric core of Peng’s six guidelines. Lower data-to-ink: gridded backgrounds, 3D effects, redundant legends, gradient bars where a dot would do. Higher data-to-ink: minimal theme, single annotated reference line, faceted small multiples without per-panel scaffolding. Peng demonstrates this with the side-by-side Chicago mortality plots: same data, but the right-hand version (line + minimalist theme) communicates the heat-wave spike instantly while the left-hand version (filled area + colored background + gridlines) buries the signal under decoration. When you’re making a plot and uncertain what to add, the right move is usually subtraction.

Color is the aesthetic with the most communicative power and the most common misuse. Three failure modes recur in student work: (1) using a qualitative palette for ordered data — Set1 for income brackets — which destroys the ordering by mapping it to perceptually unordered hues; (2) using a sequential palette for unordered data — viridis for political party — which suggests an ordering that doesn’t exist; (3) confusing color with fill — color is the edge of an element, fill is the interior, and geom_point mostly responds to color while geom_bar mostly responds to fill, so a misapplied scale function silently does nothing. The recovery: pick palette family by data type (qualitative / sequential / diverging), prefer viridis for sequential (it’s perceptually uniform and colorblind-safe), and double-encode (color + shape, color + label) for accessibility.

A small reference for the six guidelines

Each guideline paired with the ggplot2 mechanisms that implement it:

Guideline Primary mechanisms
1. High data density theme_minimal / theme_few / theme_tufte, drop gridlines, prefer dots over bars where appropriate
2. Clear labels labs(), xlab() / ylab(), coord_flip(), scales::comma, fct_recode
3. Useful references geom_smooth, geom_hline / geom_vline, geom_ribbon for normal range, alpha + gray
4. Highlighting geom_text / geom_label with subsetted data, aes(color = flag), named subsets
5. Small multiples facet_wrap (one var), facet_grid (two vars), scales = "free_y", ordered factor levels
6. Meaningful order forcats::fct_reorder, fct_infreq, arrange + factor(levels = unique(.)) pattern

Two patterns worth memorizing

The “highlight a subset” pattern. Generalized from Peng’s noteworthy-players example:

noteworthy <- data |>
  filter(value > threshold) |>
  mutate(label = paste(name, value, sep = ": "))

ggplot(data, aes(x = x, y = y)) +
  geom_point(alpha = 0.4, color = "gray60") +              # background
  geom_point(data = noteworthy, color = "firebrick",        # highlight
             size = 2.5) +
  geom_text(data = noteworthy, aes(label = label),          # label
            vjust = "inward", hjust = "inward",
            color = "firebrick", size = 3) +
  theme_few()

The structural move: two layers, two datasets. The full data forms a quiet background; the subset of interest gets stronger color, larger size, and a text label. The vjust = "inward" and hjust = "inward" keep labels inside the plot bounds without manual positioning.

The “order by another variable” pattern. Useful any time the categorical axis is alphabetically meaningless:

data |>
  group_by(category) |>
  summarize(metric = mean(value)) |>
  arrange(metric) |>
  mutate(category = factor(category, levels = unique(category))) |>
  ggplot(aes(x = metric, y = category)) +
  geom_point() +
  theme_few()

The arrange |> mutate(factor(levels = unique(.))) idiom forces ggplot’s natural axis order to follow the metric. forcats::fct_reorder(category, metric) does the same thing in one line and is the more idiomatic choice when the data is already in the form you need.

The lift from grammar to design

ICA 03 (R4DS Ch1) was about being able to make any plot at all — knowing which geom_ to call, where aes() goes, how facets work. ICA 17 is about making plots that communicate. The difference is the difference between writing a grammatical sentence and writing a sentence that lands. The same data, the same geom_, the same facet — applied with attention to data-to-ink, references, and ordering — turns a chart that’s technically correct into one that’s actually read. Peng’s chapter is essentially a translation manual between the two layers: each design rule has a small set of ggplot2 mechanisms that implement it, so the gap between “I know what I want” and “I know what to type” is small.

Why this comes after wrangling, iteration, and acquisition

ICA 17 is where the course’s threads converge. By this point you can wrangle (ICAs 06-08), iterate (ICAs 11-12), acquire from APIs / scrape / SQL (ICAs 13-16). The natural next question is: given that you can produce any chart you want from any source, how do you produce one that’s worth showing? The answer is design discipline — not artistry, but a small set of rules consistently applied. Tufte / Few / Wainer have written entire books on this; ICA 17’s prep distills the actionable subset.

A note on the larger argument

Dykes’s article opens with Hal Varian’s 2009 prediction — “the ability to take data, to be able to understand it, to process it, to extract value from it, to visualize it, to communicate it, that’s going to be a hugely important skill in the next decades.” Reading this 17 years later, the prediction is uncontroversial: every data-related job posting now lists communication. What’s still contested is how to teach it. ICA 17 takes a position: communication isn’t a soft skill bolted onto technical work, it’s a craft with rules — data-to-ink ratio, reference framing, ordered comparison, visual hierarchy, narrative arc — that can be learned the same way function design or SQL queries can. The Forbes article is the philosophy; Peng’s chapter is the syllabus.
