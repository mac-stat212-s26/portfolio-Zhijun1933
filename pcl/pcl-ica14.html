Three concepts these vignettes quietly hinge on

Simplification is what makes fromJSON() ergonomic — and it’s what occasionally bites. When you call fromJSON(url), it tries to coerce the response into the most R-friendly type: a JSON array of primitives becomes an atomic vector; an array of objects becomes a data frame; an array of arrays becomes a matrix. This works beautifully for the 80% of APIs that return regular structures, but it can surprise you when the response is almost regular — say, an array of objects where one record is missing a field, or a “string or array” union type that varies per element. The escape hatch is fromJSON(url, simplifyVector = FALSE), which returns a raw nested list, letting you handle the shape yourself with purrr/tidyr. Worth knowing that the default exists and that turning it off is the move when you hit weird structure.

Most APIs paginate, and paginating well is half the work. A single API call returning 1100 records (like the ProPublica example) is already split across 11 pages of 100 each — and most APIs are similar. The ProPublica vignette shows the canonical pattern: a for loop over page numbers, accumulating into a list, then rbind_pages (or list_rbind from purrr) at the end. The choice points: how does the API page (page number / offset / cursor)? How do you know when to stop (response count / empty page / explicit “has_more” flag)? Get those two right and the iteration becomes mechanical. Get them wrong and you either miss data or loop forever.

Auth credentials never live in your code, and .Renviron is the right place. Every example in the “Fetching JSON” vignette that requires a key shows it inline — that’s pedagogical, not safe. In real work, the API key goes in ~/.Renviron as NYT_API_KEY=abc123, accessed via Sys.getenv("NYT_API_KEY"), with .Renviron itself listed in .gitignore. The reason is asymmetric: writing api-key=hard_coded_string once costs nothing, but if you push that to GitHub, your key is now public and you have hours of revocation/rotation work ahead. Worth treating as a habit even when prototyping — the cost of doing it right is one extra line, the cost of doing it wrong is real.

A small reference for jsonlite’s type mapping

The four-cell table that the Getting Started vignette is essentially organized around:

JSON structure Example Simplifies to Argument
Array of primitives ["a", "b", "c"] atomic vector simplifyVector = TRUE
Array of objects [{"x": 1}, {"x": 2}] data frame simplifyDataFrame = TRUE
Array of arrays (same length) [[1, 2], [3, 4]] matrix simplifyMatrix = TRUE
Anything irregular [{"x": 1}, {"y": 2}] or mixed nested list use simplifyVector = FALSE

All three simplification arguments default to TRUE. Disabling them gives you raw structure to work with manually.

Two patterns worth memorizing

The paginated-list pattern. Generalized from the ProPublica example:

base <- "https://api.example.com/v1/items"

pages <- list()
for (i in 0:max_page) {
  url <- paste0(base, "?page=", i, "&", auth)
  resp <- fromJSON(url, flatten = TRUE)
  pages[[i + 1]] <- resp$items
  message("Retrieved page ", i)
  if (length(resp$items) == 0) break    # stop on empty
}

all_items <- jsonlite::rbind_pages(pages)

The purrr version is shorter and composable but functionally equivalent:

0:max_page |>
  map(\(i) fromJSON(paste0(base, "?page=", i, "&", auth), flatten = TRUE)$items) |>
  list_rbind()

For unknown total page count, replace 0:max_page with a while loop that breaks when length(items) == 0.

The store-the-key-safely pattern. Once per machine, in ~/.Renviron:

CENSUS_API_KEY=abc123def456
NYT_API_KEY=xyz789

Restart R (so the file is re-read on session start), then in any analysis:

key <- Sys.getenv("CENSUS_API_KEY")
url <- urltools::param_set(base_url, "key", key)
data <- fromJSON(url)

The .Renviron file is per-user; on a Mac it lives in ~/.Renviron and isn’t tracked by any project’s .git. If you accidentally commit a key, rotate it immediately — assume it’s compromised the moment it hits a public repo.

Why this comes after URLs and JSON

ICA 13 gave you URL structure and JSON anatomy in the abstract. ICA 14 puts both into motion: a URL is what you build to make a request, JSON is what comes back. The vignettes are deliberately concrete — six different APIs, each illustrating a different design choice — because no abstract description of “an API” can prepare you for the variety you’ll meet in the wild. By the time you finish ICA 14, the goal is that hitting a new API for the first time feels like applying a familiar template: build URL, fetch, inspect shape, flatten if needed, paginate if needed, bind. The httr2 package layered on top of all this (status codes, retries, OAuth flows, rate limiting) is incremental — once the JSON round trip is muscle memory, the request layer is just bookkeeping.

A note on what this enables

HW07 P2 is a paginated API call: one Census ACS endpoint, called once per year from 2005 to 2022, results bound into one long tibble. That’s the paginated-list pattern with years instead of page numbers. The nested response (variables and geographies in separate JSON arrays) needs unnest_wider to flatten. The API key sits in .Renviron, accessed via Sys.getenv(), never visible in the qmd. Every piece of the prep maps to a piece of the homework — and the same template will work for every JSON API you’ll encounter outside this course (OpenWeather, GitHub, Spotify, FRED, World Bank, OECD).
