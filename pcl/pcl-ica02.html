Three threads worth pulling

Why raw data must stay untouched. The discipline of “never edit a raw file” is what makes results reproducible. If population.csv is the original from the source and population-cleaned.csv is the output of population-process.qmd, anyone can re-run the script and reproduce the cleaned file. The moment you edit population.csv directly — even just to fix a typo — the lineage breaks and “what did this come from” stops being answerable. The data/raw/ + data/processed/ split is the file-system embodiment of this principle.

Why relative paths beat absolute paths. ~/Desktop/STAT212/... works on my machine and breaks on yours; ./data/file.csv works on both, because it’s defined with respect to the project root rather than the user. The same logic that makes Git portable across machines makes relative paths portable. Web URLs are the only exception — they’re already universal.

Why merge conflicts aren’t bugs but features. A merge conflict is Git refusing to silently overwrite one collaborator’s work with another’s. The conflict markers (<<<<<<< / ======= / >>>>>>>) are Git saying “I can’t decide which version is right; you have to.” Resolving a conflict is a tiny act of editorial judgment — looking at both versions, deciding which to keep (or merging them by hand), then committing the resolution. Annoying in the moment, but vastly better than the alternative of losing work.

Quick keyboard wins

The shortcuts that pay off fastest in everyday RStudio work, ranked by how often they fire:

  • Cmd Return — run current line. Replaces “click in line, click Run button.”
  • Cmd Option I — insert code chunk. Replaces typing ```{r} and the closing fence by hand.
  • Cmd Shift M — pipe |>. The pipe is the most-typed three-character sequence in tidyverse code; this shortcut alone saves real time.
  • Cmd Shift C — toggle comment on selected lines. Indispensable for “let me try this without that block running.”
  • Cmd Shift K — cheat sheet for all other shortcuts.

Portfolio inspection

The fourth piece of ICA-02 prep is opening my portfolio GitHub repo and noting how files are organized. The structure follows the conventions in the file-organization reading: a top-level Quarto book project (_quarto.yml, index.qmd, 404.qmd) with chapter directories (pv/, tt/, pcl/, ica/, extra/) as siblings, plus assets/ and style/ for shared resources, and docs/ as the rendered output. No nested repos, no raw data lives next to code, paths inside _quarto.yml are all relative — the design follows the principles the readings argue for.
