Sources of Information — What is a Data Frame in R?

The exercise is to ask the same question of seven different sources and watch how the answers differ. What follows is a compressed synthesis of what each source actually says, not a retyping of its text.

(a) What each source says

Search engines (Google). The top results are tutorial sites (W3Schools, GeeksforGeeks, DataCamp). They converge on the framing “a data frame is a table-like structure with rows and columns, where each column can have a different type.” Optimized for findability; thin on mechanism.

LLMs (Gemini). Returns the tutorial-site framing with an extra sentence or two of context — “similar to a spreadsheet or SQL table” — and, if prompted, offers a data.frame() example. Fluent and fast; confidently silent about what’s underneath the abstraction.

Intro textbooks (R4DS). Wickham & Grolemund introduce the tibble first (tbl_df) rather than data.frame, and frame it as “the modern reimagining of the data frame.” The pedagogical move is to let the learner use tidy data without worrying about legacy base-R footguns (stringsAsFactors, partial matching, drop = TRUE in [).

R Manuals (official). An Introduction to R is terser and more precise: a data frame is a list of variables of the same number of rows with unique row names, given class "data.frame". This definition is the one that tells you what the object is, not what it looks like.

Advanced textbooks (Advanced R). Wickham is explicit: a data frame is an S3 object built on top of a named list of equal-length vectors, with a row.names attribute and class = "data.frame". Everything else — the rectangular printout, [ behavior, $ access — is convention layered on top of the list substrate. This is the view that generalizes.

data.frame() documentation. The function signature tells you what the constructor actually does: recycles shorter columns, mangles or checks names, and (since R 4.0) no longer stringifies-to-factor by default. Read this when something surprising happens at construction time.

tibble class documentation. Specifies exactly how tbl_df differs from data.frame: [ always returns a tibble (never drops to a vector), no partial name matching, no row names by default, lazier coercion, nicer printing. Read this when something surprising happens downstream of a mutate() or a bracket.

(b) How the sources differ

Two axes separate them cleanly:

  1. High-level (what it looks like) vs. low-level (what it is). Search engines, LLMs, and intro textbooks stay on the first side; manuals, Advanced R, and the class documentation cross to the second.
  2. Happy-path vs. edge-case. Tutorials describe the happy path; documentation describes the behavior under unusual inputs — which is where bugs actually live.

The pattern that matters: a source that looks authoritative (a confident LLM answer, a polished tutorial site) can be shallower than a source that looks dry (the R manual page). Authority of tone is not authority of depth.

Our definition of a data frame in R

NoteOur definition

A data frame in R is an S3 object built on a named list of equal-length vectors, with a row.names attribute supplying an implicit row index and class = "data.frame" supplying method dispatch. The rectangular “table with rows and columns” is the user-facing abstraction; the list of vectors is the implementation. A tibble (tbl_df) is a refined subclass that changes defaults (no partial name matching, non-dropping [, no row names, no silent string-to-factor coercion) while preserving the list-of-vectors substrate.

A small verification pass makes this concrete:

Code
df <- data.frame(x = 1:3, y = c("a", "b", "c"))

class(df)          # "data.frame"
[1] "data.frame"
Code
typeof(df)         # "list" — this is the move
[1] "list"
Code
is.list(df)        # TRUE
[1] TRUE
Code
length(df)         # 3 columns — same as length of the underlying list
[1] 2
Code
attributes(df)     # names, class, row.names
$names
[1] "x" "y"

$class
[1] "data.frame"

$row.names
[1] 1 2 3

typeof(df) == "list" is the sentence that turns “data frame” from a black box into a navigable object. Once you know the substrate is a list, every list operation — [[, $, lapply(), purrr::map() — is available, and the rectangular shape is just a display convention the class enforces on top.

Code
# A tibble behaves like a data frame but tightens the defaults
tb <- tibble(x = 1:3, y = c("a", "b", "c"))

class(tb)          # "tbl_df" "tbl" "data.frame" — inheritance chain
[1] "tbl_df"     "tbl"        "data.frame"
Code
identical(as.list(tb), as.list(df))   # same substrate, different defaults
[1] TRUE

Levels of Understanding

Two levels:

  • High-level (intuitive): enough to implement common tasks. “A data frame is a table with columns of possibly different types.”
  • Low-level (foundational): enough to tackle new problems and invent new solutions. “A data frame is an S3 object over a named list of equal-length vectors; [ dispatches through the data.frame method; tibbles are a subclass with tightened defaults.”

The class standard is low-level. The practical workflow is to start high-level (LLM, tutorial, R4DS) to get oriented, then dig into documentation and Advanced R to actually understand why the thing behaves the way it does. The high-level pass tells you which questions to ask; the low-level pass gives answers that generalize.

A compact test for whether I have reached the low-level understanding: can I predict what an unfamiliar operation will do to the object before I run it, by reasoning about its structure? If yes, low-level. If I have to run it to see, still high-level.

Data Wrangling Verbs — Review from COMP/STAT 112

Organized by what they do to the shape of the data, not in alphabetical order — grouping by effect is how I actually recall them under pressure.

Column operations

  • mutate() — creates or modifies columns.
  • select() — keeps a subset of columns.

Row operations

  • filter() — keeps a subset of rows by condition.
  • arrange() — sorts rows.

Group-and-collapse

  • group_by() — internally groups rows by one or more columns (no visible change until a downstream verb consumes the grouping).
  • summarize() — collapses each group to a single row using n(), sum(), mean(), min(), max(), median(), sd(), etc.
  • count() — shortcut for group_by(...) |> summarize(n = n()).

Joins — combining two tables

Mutating joins (can add columns):

  • left_join() — keep all rows in the left table.
  • full_join() — keep all rows in both.
  • inner_join() — keep only rows with matches in both.

Filtering joins (never add columns, only filter the left table):

  • semi_join() — keep left-table rows that have a match on the right.
  • anti_join() — keep left-table rows that have no match on the right.

Reshape — change the rectangle itself

  • pivot_wider() — one column’s values become new column names; another column’s values populate them. Used when the tidy long form needs to be read by a human or by a wide-format model.
  • pivot_longer() — the inverse: many columns collapse into a name column and a value column. Used to get data into tidy form from spreadsheet-shaped inputs.

A compact demonstration

One chained pipeline exercising most of the verbs, for muscle memory:

Code
mtcars |>
    rownames_to_column("car") |>
    mutate(power_to_weight = hp / wt) |>
    filter(cyl %in% c(4, 6, 8)) |>
    group_by(cyl) |>
    summarize(
        n          = n(),
        mean_mpg   = mean(mpg),
        mean_ptw   = mean(power_to_weight),
        .groups    = "drop"
    ) |>
    arrange(desc(mean_mpg))
# A tibble: 3 × 4
    cyl     n mean_mpg mean_ptw
  <dbl> <int>    <dbl>    <dbl>
1     4    11     26.7     37.9
2     6     7     19.7     39.9
3     8    14     15.1     53.9

One sentence about each verb in the pipeline: rownames_to_column() moves the implicit row-name index into a real column (because row names are a legacy feature that tibbles drop); mutate() adds a derived column; filter() subsets rows; group_by() sets the grouping for the next verb; summarize() collapses each group to one row; arrange(desc(...)) sorts descending. The pipeline is the mental model: shape the columns, shape the rows, group, collapse, sort.

Reflection

Two things I want to carry forward from this first lesson.

Source selection is a skill, not a preference. The naïve model is “use the best source.” The real model is which source answers which kind of question. A tutorial site answers how do I do X right now?; an LLM answers give me a fluent overview of X; the R manual answers what does X actually do under unusual inputs?; Advanced R answers why does X behave the way it does structurally?. A workflow that always reaches for the same source is a workflow that will be wrong in predictable ways — too shallow when precision matters, too dense when orientation is what’s needed. The cost of the wrong source is time: shallow sources fail silently on edge cases, dense sources eat hours on questions that had a one-sentence answer.

Low-level understanding is what lets code generalize. The typeof(df) == "list" line is the entire game. If I know a data frame is a list of vectors, purrr::map_dfc(), [[, bracket subsetting, and every coercion surprise become one system rather than five disconnected tricks. The high-level “table with columns” mental model is sufficient for mutate() and filter(); it’s insufficient the moment I need to write a function that takes any tabular object and does something sensible with it. This is the same instinct that sits behind the str() pass in later lessons — the object is a list underneath, so open it and look.
